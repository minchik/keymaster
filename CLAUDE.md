Keymaster is a macOS Swift CLI that stores/retrieves Keychain secrets guarded by Touch ID **or a paired Apple Watch** (the access-control object is `[.biometryAny, .or, .companion]`; no passcode fallback). It is built as a **signed `.app`**, not a bare `swiftc` binary, because the biometric guard needs the restricted `keychain-access-groups` entitlement (AMFI kills an unsigned/unprovisioned binary that carries it). The program is a CLI: its entry point in `keymaster/keymaster/keymasterApp.swift` does the Keychain work and `exit()`s before any AppKit run loop starts, so no window is shown. The code is split behind a protocol seam so the security-critical logic is unit-testable:

- `keymaster/keymaster/keymasterApp.swift` тАФ the `@main` CLI surface only: the ArgumentParser command structs (the namespace-**independent** top-level `get`/`run` plus `version`, the `secret set`/`get`/`rm`/`ls` group, and the `oauth set`/`get`/`rm`/`ls` group) plus stdin/version/IO glue (`readSecret`, the no-echo `readLineWithEchoOff` TTY reader, `fail`, `marketingVersion`) and the shared `warnStaleRefreshToken` helper. Top-level `get` is the smart resolver (classify plain-vs-OAuth under one approval, read or mint); `secret get` is a pure plain read; `secret set` and `oauth set` each call the shared `storeSecret(..., in:conflictingWith:)`, which **refuses** (writes nothing) if the name already lives in the other namespace тАФ a best-effort guard (not atomic against a concurrent conflicting writer) keeping a name in one store. `Secret.List`/`OAuth.List` (both `commandName: "ls"`, no arguments) build `SecretManager(backend: SystemKeychain(), namespace: тАж)`, call `list(reason:)`, and print each sorted name on its own line (empty store prints nothing). Each command builds a `SecretManager(backend: SystemKeychain())` (or an `OAuthManager(backend: SystemKeychain(), exchanger: URLSessionTokenExchanger())`) and maps `KeychainError` тЖТ `fail(error.message)`.
- `keymaster/keymaster/SecretManager.swift` тАФ **Foundation-only**, no `Security`/`LocalAuthentication`/`OSStatus` in any signature: the `KeychainError` enum (including the `crossNamespaceConflict(name:existsIn:)` case, whose `message` names the existing item's kind and the exact `secret rm`/`oauth rm` command), the `KeychainNamespace` enum (`secret`/`oauth`), the `KeychainBackend`/`AuthSession` protocols (namespace-aware `add`/`read`/`delete`/`read(using:)` plus a no-prompt **fail-closed** `exists` probe (`throws -> Bool`: `false` only for `errSecItemNotFound`, throws on any other status so a transient keychain error is never silently read as "absent") used by the cross-namespace guard, and the **session-aware** `exists(key:using:namespace:)`/`update(key:secret:using:namespace:)` pair that classify-probes and writes back through a caller's single Touch ID approval тАФ the former with the same fail-closed contract, the latter the in-place rotation write-back тАФ plus the session-aware `list(using:namespace:)` name enumeration: a bare metadata listing (no values, no decryption) that never prompts on its own and rides the caller's single approval), the testable `SecretManager` orchestration (upsert ordering, read-before-delete; its `namespace` defaults to `.secret`; and `list(reason:)`, whose **authenticate-first** ordering **is** the gate тАФ a bare metadata enumeration never decrypts so the item ACL is never evaluated, so `authenticate(reason:)` runs *before* any name is read and a cancelled/failed approval discloses nothing тАФ then returns the names `.sorted()` for deterministic output), the `storeSecret(_:name:in:conflictingWith:backend:)` write seam (a write-time NUL guard тАФ `guard !secret.contains(0)` тАФ runs **first**, refusing to store a value with an embedded NUL that `get`/`run`'s `decodeEnvValue` would reject on read, so a bad value triggers no keychain I/O; then the refuse-on-conflict primitive runs a no-prompt `exists` probe of the other namespace; on a hit it throws `.crossNamespaceConflict` and writes nothing, on a transient probe error it propagates that error and writes nothing, otherwise it delegates to the namespaced upsert), and the `decodeSecret`/`decodeEnvValue` UTF-8/NUL helpers (the read-time NUL check now defense-in-depth for legacy/out-of-band bytes the write-time guard never saw). Compiled into the app **and** the test bundle.
- `keymaster/keymaster/SystemKeychain.swift` тАФ app-target only: the real `KeychainBackend` adapter wrapping the `SecItem*`/`LAContext` syscalls, the access-control/access-group constants (`makeAccessControl()` builds the sole `SecAccessControl` with `[.biometryAny, .or, .companion]` тАФ Touch ID **or** a paired Apple Watch, still no `.userPresence`/`.devicePasscode` fallback and no reuse window), the `KeychainNamespace` тЖТ service-prefix mapping (`dev.mnck.` / `dev.mnck.oauth.`) and matching per-namespace account mapping (`keymaster` / `keymaster.oauth`, distinct so the prefix-overlapping namespaces can never alias), the `OSStatus` тЖТ `KeychainError` mapping, and the `list(using:namespace:)` name enumeration (a **fresh** query, **not** the single-key `baseQuery`: `kSecClassGenericPassword` + the per-namespace `kSecAttrAccount` + `kSecMatchLimitAll` + `kSecReturnAttributes:true`/`kSecReturnData:false` so nothing is decrypted, with `kSecUseDataProtectionKeychain:true` kept тАФ without it the query hits the legacy file keychain and silently returns `[]` тАФ and the session's `kSecUseAuthenticationContext`; `errSecItemNotFound` тЖТ `[]`, then it keeps only items whose `kSecAttrService` carries the namespace's own `servicePrefix` and strips that prefix to recover each name, so a plain key literally named `oauth.Foo` round-trips). Unit-untestable (needs signing + Touch ID hardware), so it is covered by the manual checklist below.
- `keymaster/keymaster/RunSupport.swift` тАФ the pure, biometric-free helpers for the `run` subcommand (`parseKeyMapping`, `mergedEnvironment`, `runProcess`), also Foundation-only and unit-tested headlessly.
- `keymaster/keymaster/TerminalInput.swift` тАФ **Foundation-only** pure no-echo line-editing reducer extracted from the TTY reader: the `NoEchoLineEvent` enum (`continue`/`endLine`/`endInput`) and `noEchoLineEvent(after:into:)` (LF/CR тЖТ `endLine`; `0x04` Ctrl-D тЖТ `endInput` if the buffer is empty else `endLine`; `0x7F`/`0x08` тЖТ `eraseLastScalar` (the last UTF-8 scalar: pop the last byte, then keep popping continuation bytes `b & 0xC0 == 0x80` up to a 4-byte cap, so a backspace inside a multi-byte character never leaves a dangling partial sequence that would decode to U+FFFD); `0x15` Ctrl-U тЖТ kill the whole line (canonical VKILL); `0x17` Ctrl-W тЖТ `eraseLastWord` (canonical VWERASE: drop trailing blanks, then back to the previous SPACE/TAB or line start тАФ scalar-safe because blanks are ASCII and never appear inside a multi-byte sequence); else append). Ctrl-U/Ctrl-W are re-implemented here because clearing `ICANON` (to drop the `MAX_CANON` cap) also disables the kernel's VKILL/VWERASE editing, so without these cases those bytes would be silently appended into the echo-off secret. The termios setup + per-byte `read()` loop in `readLineWithEchoOff` is the thin, manual syscall wrapper around this reducer; the reducer holds the unit-tested logic. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/OAuthRecord.swift` тАФ **Foundation-only** OAuth data model: the tolerant `Codable` `OAuthRecord` (snake_case `token_endpoint`/`client_id`/`client_secret?`/`refresh_token`/`scopes?`) with `encoded()`/`validate()` (required fields non-empty, no field may contain an embedded NUL byte, and `token_endpoint` must be `https`; the NUL check closes the escaped-NUL bypass тАФ `JSONEncoder` escapes U+0000 to a six-character text escape, so `storeSecret`'s byte-level `contains(0)` guard never sees it тАФ that would otherwise let `oauth set` persist a silently-bricked credential), and the `TokenResponse` reply type (only `access_token` + optional `refresh_token`; the unused `token_type`/`expires_in` were dropped so a non-conformant value for either тАФ e.g. a string `expires_in` тАФ can no longer break the decode, and a custom `init(from:)` decodes `refresh_token` **tolerantly** тАФ a present-but-non-string value, e.g. a JSON number, is treated as "no rotation" rather than failing the whole decode, so it can't mask a valid `access_token` either; `access_token` stays strictly required). Compiled into the app **and** the test bundle.
- `keymaster/keymaster/TokenExchanger.swift` тАФ **Foundation-only** token-exchange seam: the `TokenExchanger` protocol plus the pure `buildTokenRequest` (form-encoded RFC 6749 ┬з6 POST) and `parseTokenResponse` (2xx decode of `access_token` + optional `refresh_token` тАФ unknown keys like `token_type`/`expires_in` are ignored тАФ with a non-empty `access_token` check and a **NUL-byte rejection on both** `access_token` and the rotated `refresh_token` (the latter symmetric so a NUL credential can never be persisted by the rotation write-back, which bypasses `storeSecret`'s guard and would survive it anyway since `JSONSerialization` escapes U+0000); RFC `{error,error_description}` mapping incl. the `invalid_grant` re-auth hint) helpers. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/OAuthManager.swift` тАФ **Foundation-only** OAuth orchestration shared by `get` and `run`: `OAuthManager` over the `KeychainBackend`+`TokenExchanger` seams. `get` and `run` funnel through one **authenticate-first** `resolve` core that calls `authenticate(reason:)` **once** (the single Touch ID prompt) and then does everything else through that session тАФ the namespace **classify** (`resolveNamespace(name:using:)` тЖТ the session-aware `exists(using:)`), the record/secret **read** (`read(using:)`), and the rotation **write-back** (`update(using:)`) тАФ so all three ride the one approval instead of prompting separately. The two wrappers are `resolveRunEnvironment(mappings:reason:)` (the `run` batch тЖТ `(env, staleKeys)`) and `resolveSecret(name:reason:)` (the `get` single name тЖТ `(value, refreshTokenStale)`); `mint(name:using:)` reads+exchanges an OAuth record under the supplied session, and `finish(name:recordData:session:)` decodes тЖТ exchanges тЖТ conditionally writes back. The write-back re-asserts the NUL guard at its own boundary тАФ a rotated `refresh_token` containing a NUL is **not** persisted (skipped, flagged stale, the just-minted token still used), defense-in-depth in case a `TokenExchanger` conformer other than the shipping one doesn't reject it first. The rotation write-back is **lossless**: it edits `refresh_token` in place in the raw stored JSON object (`JSONSerialization`) rather than re-encoding the 5-field `OAuthRecord` model, so any extra keys an out-of-band write stored survive a rotation (the rotated bytes are decode-equivalent, not byte-canonical тАФ `JSONSerialization` escapes the `token_endpoint` slashes). `resolveNamespace` is **fail-closed** (`throws`): it propagates a transient `exists(using:)` error from either namespace probe instead of falsely returning `nil` (no record found). Classifying **after** the prompt is the intended **security property**, not a cost: keymaster never discloses whether a name exists without a biometric approval first, so unauthenticated key-existence probing is impossible тАФ a name in **neither** store deliberately aborts *after* the single prompt (a key-prefixed "not found"), not before it, so the name's absence is never leaked pre-approval. The `MintResult` type carries the minted access token plus the stale-refresh-token flag. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/URLSessionTokenExchanger.swift` тАФ app-target only: the real `TokenExchanger` doing the synchronous `URLSession` POST (semaphore-gated, with a timeout) and casting to `HTTPURLResponse` for `parseTokenResponse`. It uses a **dedicated** `URLSession` (not `URLSession.shared`) with a no-redirect `URLSessionTaskDelegate` that **refuses every HTTP redirect** (returns `nil` from `willPerformHTTPRedirection`), so the credential-bearing POST body (the `refresh_token`, and any `client_secret`) is never resent to a redirect target; a redirecting endpoint surfaces as a request error. Unit-untestable (needs the network), so тАФ like `SystemKeychain.swift` тАФ it stays out of the test bundle and is covered by the live-provider smoke test below.

### Dependency

Argument parsing uses [swift-argument-parser](https://github.com/apple/swift-argument-parser) тАФ each subcommand (the top-level `get`/`run` plus the `secret set`/`get`/`rm`/`ls` and `oauth set`/`get`/`rm`/`ls` groups) is a `ParsableCommand` in `keymasterApp.swift`; ArgumentParser is imported only there (the `SecretManager`/`SystemKeychain` layers are ArgumentParser-free). It is wired in as an SPM package and **resolved automatically**: Xcode fetches it on first build, and the release workflow's `xcodebuild archive` fetches it on a clean runner (no `-disableAutomaticPackageResolution`). The pinned version lives in `keymaster.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved`, which is **committed** so CI builds are reproducible rather than re-resolving to the latest `1.x`. The library links statically into the binary, so there is no extra framework to sign or notarize. If you ever re-add the package by hand, link **only** the `ArgumentParser` library product to the **keymaster** target тАФ leave the `generate-manual` and `generate-docc-reference` executable products set to **None**.

A plain `swiftc keymaster*.swift` build will **not** work: the biometric `kSecAttrAccessControl` requires the `keychain-access-groups` entitlement, an unsigned binary that adds without it gets `errSecMissingEntitlement` (-34018), and a binary that carries it without a provisioning profile is SIGKILLed by AMFI. The `.app` target is what supplies the profile.

## Testing

The raw keychain/biometric/exec/network syscalls (`SystemKeychain.swift`, `URLSessionTokenExchanger.swift`) can't run headless or in CI тАФ verify them by hand on a Touch ID Mac (below). The Foundation-only logic in `SecretManager.swift`, `RunSupport.swift`, `OAuthRecord.swift`, `TokenExchanger.swift`, `OAuthManager.swift`, and `TerminalInput.swift` **is** unit-tested via the `keymasterTests` target (Swift Testing тАФ `import Testing`, `@Test`/`#expect`). `SecretManager`'s ordering and decode invariants (upsert `add`тЖТreadтЖТdeleteтЖТadd, read-before-delete on `remove`, the authenticate-before-list `ls` flow тАФ sorted/namespace-scoped output, empty store, single-session reuse, and the no-disclosure-on-auth-failure invariant тАФ the `run` batch flow, UTF-8/NUL rejection) are exercised against a `FakeKeychainBackend` (a namespace-aware, dictionary-backed `KeychainBackend` that records call order тАФ including `listUsing` тАФ fakes `list(using:namespace:)` (returning the namespace's keys unsorted so the orchestration's sort is load-bearing), and can throw programmed `KeychainError`s); the OAuth mint/rotation/classification/run-resolver flow is exercised against the same fake plus a `FakeTokenExchanger` (a programmable `TokenResponse`-or-error double), so the security-critical logic is covered without any real Keychain or network access:

```bash
xcodebuild test -project keymaster/keymaster.xcodeproj -scheme keymaster \
  -destination 'platform=macOS' -only-testing:keymasterTests
```

The `-only-testing:keymasterTests` scope is required: it excludes `keymasterUITests`, which launches the CLI app that `exit()`s immediately and would otherwise hang/fail the run. The `keymasterTests` bundle is **host-less** (`TEST_HOST`/`BUNDLE_LOADER` cleared) because the `@main` CLI exits before an app-hosted runner can attach; the Foundation-only files тАФ `RunSupport.swift`, `SecretManager.swift`, `OAuthRecord.swift`, `TokenExchanger.swift`, `OAuthManager.swift`, and `TerminalInput.swift` тАФ are compiled into the bundle via synchronized-group membership exceptions (there is no `@testable import`), which is why each is kept Foundation-only with no `@main`/ArgumentParser/keychain (`Security`/`LocalAuthentication`)/`URLSession`-syscall symbols. `SystemKeychain.swift` and `URLSessionTokenExchanger.swift` are deliberately **not** added to the membership exceptions тАФ they hold the un-testable syscalls (keychain/biometric and network, respectively) and stay out of the test bundle.

Manual checks (need a physical Touch ID Mac and the signed `.app`):

- `secret set <newkey>`: stores the secret, **no** Touch ID prompt (the ACL is evaluated on access, not on creation).
- `secret get <key>`: a pure plain read тАФ the Keychain challenges Touch ID; the prompt names the key. A name that is actually an OAuth record returns a plain-namespace "not found" (no mint, no OAuth lookup).
- `get <key>` (the namespace-**independent** top-level resolver): **exactly one** prompt; prints the plain value or, if the name is an OAuth record, a freshly-minted access token.
- `secret rm <key>` and overwrite (`secret set` on an existing key): both prompt. `SecItemDelete` does not decrypt the item, so the biometric ACL would not challenge it on its own тАФ `SecretManager` forces an authenticated `backend.read(...)` first and proceeds to `delete` only if it succeeds (this read-before-delete ordering is the invariant covered by `SecretManagerTests`).
- `run --key A --key B -- <command>`: **exactly one** Touch ID prompt, whose text names every key and the program, unlocks all the secrets. A missing/unreadable key aborts before exec, naming the key and running nothing; cancelling the prompt also runs nothing.
- `secret ls` / `oauth ls`: each prompts **exactly once** (Touch ID or Apple Watch) and prints its own namespace's names, one per line, sorted (`secret ls` shows plain secrets only, `oauth ls` OAuth records only тАФ a plain key named `oauth.Foo` shows under `secret ls`, never `oauth ls`). An empty namespace still prompts and prints nothing (exit 0). **Cancelling the prompt lists nothing** тАФ no name is disclosed. The gate is the explicit `authenticate(reason:)`: a `kSecReturnData:false` metadata enumeration never decrypts an item, so the ACL would never challenge the listing on its own.
- **Apple Watch approval (needs a paired, unlocked, on-wrist watch):** on any prompt (`secret get`/`secret rm`/overwrite `secret set`/`secret ls`/`oauth ls`/top-level `get`/`run`/`oauth get`/minting `get`) a nearby unlocked paired Apple Watch can approve via a **side-button double-click** in place of Touch ID тАФ the ACL is `[.biometryAny, .or, .companion]`. With the watch locked / off-wrist / out of range the prompt silently falls back to Touch ID only (no spurious abort). **Items created before this ACL change keep their Touch-ID-only ACL** (the ACL is baked in at creation and never rewritten on read) until re-stored via `secret set`/`oauth set`; re-`secret set` any test keys to pick up the watch option.

OAuth manual checks (also need the network for the live exchange):

- for a minting `get`/`run`, the namespace **classify** and the `SecItemUpdate` rotation write-back both ride the single `authenticate` approval (they attach `kSecUseAuthenticationContext`), so neither adds a Touch ID prompt and the write-back stays atomic. (The no-session `exists` probe used by `secret set`/`oauth set`'s cross-namespace guard is still silent on its own.)
- OAuth-namespace reads **do** prompt тАФ **exactly once** each: `oauth get <name>` (a pure read), `keymaster get <oauth-name>` (minting: classify + read + rotation write-back all under the one prompt), and `run` with an OAuth key.
- `oauth set <name>` interactive prompts: no-echo on `client_secret` + `refresh_token`, echo for the rest; the no-echo fields read in **non-canonical** mode, so a pasted credential **over 1024 bytes** is no longer beeped/truncated (the `MAX_CANON` line limit is gone), while backspace (DEL/Ctrl-H), **Ctrl-U (kill line)**, and **Ctrl-W (erase word)** all still edit and Ctrl-C still aborts (the reducer re-implements VKILL/VWERASE that clearing `ICANON` would otherwise have dropped). The non-TTY `OAuthRecord`-JSON-from-stdin path works.
- suspend/resume a no-echo prompt: at a `secret set` or `oauth set` no-echo prompt, press **Ctrl-Z** to suspend then `fg` to resume тАФ the prompt **resumes** (the signal-interrupted `read()` retries on `EINTR`) rather than aborting with "could not read terminal input". (Only the EINTR retry is restored here; re-asserting raw/no-echo termios across the suspend is out of scope, matching `readLine()`'s behavior.)
- `oauth set` over an **existing OAuth record** prompts Touch ID (the upsert's read-before-overwrite).
- `keymaster get <oauth-name>` prints **only** the token on stdout (so `$(keymaster get X)` is clean); the stale-refresh-token warning goes to stderr.
- `keymaster run --key A --key B=oauthkey -- cmd` with a mixed plain+OAuth batch unlocks under **exactly one** Touch ID prompt naming all keys + the program; a missing key (in neither namespace) is caught by the post-prompt classify, so it deliberately aborts **after** that single prompt тАФ naming the key and running nothing. Aborting *after* the prompt is intended: a name's existence is never disclosed without a biometric approval first.
- cross-namespace **refusal**: `oauth set` over an existing plain secret (and `secret set` over an existing OAuth record) **aborts**, writing nothing and showing **no `y/N` prompt and no Touch ID** тАФ the error names the existing item's kind and the exact `secret rm`/`oauth rm` command to remove it first. Behaves identically on a TTY and piped (`printf '%s' '{тАж}' | keymaster oauth set NAME` / `printf %s "$SECRET" | keymaster secret set NAME`, where `NAME` already lives in the other namespace). After `keymaster secret rm NAME` (or `oauth rm NAME`), the same `secret set`/`oauth set` succeeds as a first-create (no prompt).
- `keymaster get <name>` error paths are **key-prefixed** by design: `get` over an OAuth record with a revoked refresh token (and over a plain secret whose stored bytes fail to decode) prints an error beginning `"<name>: тАж"` to stderr, matching `run`'s resolver prefix and `get`'s own missing-key branch. The deliberate, uniform `"<name>: тАж"` wording (incl. the post-prompt "not found") is intentional so a script can attribute the failure to the key it asked for.
- a rotating provider's new `refresh_token` is persisted (a second `run`/`get` still works); an expired/revoked refresh token surfaces the `invalid_grant` тЖТ "re-run oauth set" message.
- a token endpoint that responds with a 3xx redirect is **not** followed: the credential-bearing POST body (the `refresh_token`, and any `client_secret`) is **not** re-sent to the redirect target, and the request surfaces as an error (`get`/`run` fail) rather than minting a token.

Items live in the data-protection keychain (`kSecUseDataProtectionKeychain`), scoped to the access group from the entitlement. The legacy `security` CLI cannot see them, so a `security find-generic-password` "item not found" is expected and is **not** evidence of a bypass; the meaningful check is that `get` (and OAuth minting) requires Touch ID.

## Usage

The secret is read from **stdin**, never passed as an argument (an argument would leak via `ps` and shell history, CWE-214):

```
keymaster secret set <key>     # store a plain secret read from stdin; prompts on overwrite, not on first create
keymaster secret get <key>     # retrieve a plain secret only (no minting, no OAuth lookup), gated by Touch ID
keymaster secret rm <key>      # remove a plain secret, gated by Touch ID
keymaster secret ls            # list stored plain-secret names (one per line, sorted), gated by Touch ID
keymaster get <key>            # namespace-independent: retrieve a plain secret, OR mint an access token for an OAuth record, gated by Touch ID
keymaster run --key <NAME|ENV=key> ... -- <command [args...]>  # inject secrets/tokens, one Touch ID prompt
keymaster oauth set <name>     # store an OAuth refresh-token record (prompts on a TTY, OAuthRecord JSON on stdin otherwise)
keymaster oauth get <name>     # print the stored OAuth record's JSON, gated by Touch ID
keymaster oauth rm <name>      # remove an OAuth record, gated by Touch ID
keymaster oauth ls             # list stored OAuth record names (one per line, sorted), gated by Touch ID
```

```bash
printf %s "$SECRET" | keymaster secret set MyKeyName   # piped
keymaster secret set MyKeyName                          # interactive no-echo prompt
```

The interactive no-echo prompt (used by `secret set` and by `oauth set`'s `client_secret`/`refresh_token` fields) reads in **non-canonical** mode (echo and `ICANON` both off), so there is no `MAX_CANON` 1024-byte line cap тАФ a long pasted secret/credential is entered in full rather than beeped and silently truncated.

`secret ls`/`oauth ls` list the names stored in each namespace (one per line, `.sorted()`), gated by a **single** Touch ID / Apple Watch approval: `SecretManager.list(reason:)` authenticates *before* enumerating, because a `kSecReturnData:false` metadata query never decrypts and so never challenges the ACL on its own тАФ cancelling the prompt discloses nothing, and an empty namespace prints nothing.

`run` injects one or more keychain secrets into a child process as environment variables, unlocking the whole batch with a single Touch ID prompt. `--key NAME` injects env var `NAME` from keychain key `NAME`; `--key ENVNAME=keychainkey` injects env var `ENVNAME` from keychain key `keychainkey` (split on the first `=` only). Each name is classified (plain vs OAuth) by an `exists` probe that runs *through* the single approval (after the one prompt) тАФ running the classify under the approval is the intended security guarantee, so a name's existence is never disclosed without a biometric approval first тАФ so the batch aborts naming the key (a deliberate post-prompt "not found") if any name is in neither namespace; OAuth keys are minted into a fresh access token under the same single prompt. Any missing/unreadable key aborts before the command runs, and the child's exit code is forwarded (a signal death as `128 + signo`).

### OAuth refresh-token records

An OAuth record stores a long-lived refresh-token credential (behind Touch ID) and `get`/`run` exchange it for a short-lived **access token** on demand via the RFC 6749 ┬з6 refresh-token grant тАФ no manual copying of expiring tokens.

```bash
# store a record: on a TTY each field is prompted (no-echo for client_secret + refresh_token); otherwise read OAuthRecord JSON from stdin
keymaster oauth set GitHub
printf '%s' '{"token_endpoint":"https://example.com/token","client_id":"abc","refresh_token":"r0","scopes":"repo"}' | keymaster oauth set GitHub

keymaster get GitHub                       # Touch ID тЖТ mints + prints ONLY the access token on stdout (warnings go to stderr)
keymaster run --key TOKEN=GitHub -- ./deploy  # injects a freshly-minted TOKEN, one prompt
```

The `token_endpoint`, `client_id`, and `refresh_token` fields are required (validated on `oauth set`: non-empty, and `token_endpoint` must be `https`); no field may contain an embedded NUL byte (a JSON-escaped NUL slips past the raw-byte store guard, so `validate()` rejects it explicitly); `client_secret` (omit/empty = public client) and `scopes` are optional. keymaster never caches an access token тАФ it mints fresh each time. If the provider rotates the refresh token (returns a new `refresh_token`), the stored record is updated in place atomically; the write-back is **lossless** тАФ it edits `refresh_token` in the raw stored JSON rather than re-encoding the modelled fields, so any extra keys survive (the rotated bytes are decode-equivalent, not byte-canonical). If that write-back fails, the just-minted token is still used and a warning naming the key (re-run `oauth set`) is printed to **stderr** (non-fatal). An expired/revoked refresh token surfaces an `invalid_grant` тЖТ "re-run oauth set" message. The token exchange uses a dedicated session that **refuses HTTP redirects**, so the credential-bearing POST is never resent to a redirect target (a redirecting endpoint fails the request instead).

## Storage

Plain secrets are stored in the data-protection keychain under service `dev.mnck.<key>`, account `keymaster`, access group `<TeamID>.dev.mnck.keymaster`, accessibility `kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly`, and a `[.biometryAny, .or, .companion]` access control (Touch ID **or** a paired Apple Watch; deliberately **no** `.userPresence`/`.devicePasscode` fallback). The ACL is baked into each item at creation and is never rewritten on read, so items stored before this watch-capable ACL keep their original Touch-ID-only ACL until re-stored via `secret set`/`oauth set` (informational; decode-irrelevant).

OAuth records live in a **separate namespace** under service `dev.mnck.oauth.<name>` and a **distinct account** `keymaster.oauth` (same access group / accessibility / access control), holding the whole credential as canonical JSON. The distinct account matters: the plain service prefix `dev.mnck.` is a textual prefix of the OAuth one `dev.mnck.oauth.`, so a plain key like `oauth.<name>` would otherwise compute the same `kSecAttrService` as OAuth record `<name>`; since the keychain primary key is the (service, account) pair, the per-namespace account keeps the two stores disjoint for **all** key text. Each name lives in **exactly one** namespace: `get`/`run` resolve it unambiguously via an `exists` probe that rides their single Touch ID approval (the session-aware `exists(using:)`, run *after* the one prompt; `.oauth` wins if a name defensively exists in both), while `secret set`/`oauth set`'s cross-namespace guard uses the no-prompt `exists`. The two creators тАФ plain `secret set` and `oauth set` тАФ guard cross-namespace collisions by **refusing**: each runs a no-prompt `exists` probe of the *other* namespace before writing, and if the name already exists there `storeSecret` throws `.crossNamespaceConflict` and writes nothing (its message names the existing item's kind and the exact `secret rm`/`oauth rm` command). The probe is **fail-closed**: a transient keychain error during it makes `secret set`/`oauth set` refuse (propagate the error) rather than silently skip the refusal тАФ so an undetermined keychain state can never let the same name end up in **both** stores. The refusal costs no Touch ID prompt; to replace a credential the user must `secret rm`/`oauth rm` the old item first, then create the new one. This refusal is **best-effort, not atomic**: the probe and the write are two separate Keychain operations and SecItem offers no cross-(service,account) atomic check-and-add, so two *concurrent* conflicting writers (a user racing `keymaster secret set K` against `keymaster oauth set K`) could each probe the other store as absent and both write, leaving `K` in both. This is an accepted edge case for a single-user Touch ID CLI тАФ the same class of concurrent-writer race the upsert's non-atomic addтЖТreadтЖТdeleteтЖТadd already accepts (see the `.duplicate` note); the worst case is benign (`get`/`run` still resolve deterministically тАФ `.oauth` wins the tie тАФ and either `secret rm`/`oauth rm` clears it), and a post-write rollback would only move the window while risking mutual deletion. So absent a concurrent conflicting writer a name lives in exactly one store.
