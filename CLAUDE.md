If a file is ignored it is ignored on purpose — `.bin/`, `.envrc`, `.claude/`, `.ralphex/`, `docs/`, `build/`, and the Xcode/coverage artifacts must stay untracked. If you genuinely think an ignored file needs to be committed, **stop and ask first** — never override the ignore yourself.

Keymaster is a macOS Swift CLI that stores/retrieves Keychain secrets guarded by Touch ID **or a paired Apple Watch** (the access-control object is `[.biometryAny, .or, .companion]`; no passcode fallback). It is built as a **signed `.app`**, not a bare `swiftc` binary, because the biometric guard needs the restricted `keychain-access-groups` entitlement (AMFI kills an unsigned/unprovisioned binary that carries it). The program is a CLI: its entry point in `keymaster/keymaster/keymasterApp.swift` does the Keychain work and `exit()`s before any AppKit run loop starts, so no window is shown. The code is split behind a protocol seam so the security-critical logic is unit-testable:

- `keymaster/keymaster/keymasterApp.swift` — the `@main` CLI surface only: the ArgumentParser command structs (the **namespace-explicit** top-level `get`/`run` — each takes a `secret.`/`oauth.`-prefixed key — plus `version`, the `secret set`/`get`/`rm`/`ls` group, and the `oauth set`/`get`/`rm`/`ls` group) plus stdin/version/IO glue (`readSecret`, the no-echo `readLineWithEchoOff` TTY reader, `fail`, `marketingVersion`, and `execCommand` — the app-only `execve` syscall wrapper that `run` uses to **replace** this process with the command, so the child inherits keymaster's PID/foreground process group/controlling terminal and interactive TUIs work; un-testable like the other raw-syscall wrappers, covered by the manual checklist) and the shared `warnStaleRefreshToken` helper. Top-level `get` parses the `secret.`/`oauth.` prefix in `validate()` (before any prompt), then under one approval reads (`.secret`) or mints (`.oauth`) the **de-prefixed** bare key; `secret get` is a pure plain read; `secret set` and `oauth set` each call the shared `storeSecret(_:name:in:backend:)`, a plain upsert with only the write-time NUL guard — duplicates across namespaces are allowed, so the same name may live in **both** stores and neither creator consults the other. `Secret.List`/`OAuth.List` (both `commandName: "ls"`, no arguments) build `SecretManager(backend: SystemKeychain(), namespace: …)`, call `list(reason:)`, and print each sorted name on its own line (empty store prints nothing). Each command builds a `SecretManager(backend: SystemKeychain())` (or an `OAuthManager(backend: SystemKeychain(), exchanger: URLSessionTokenExchanger())`) and maps `KeychainError` → `fail(error.message)`.
- `keymaster/keymaster/SecretManager.swift` — **Foundation-only**, no `Security`/`LocalAuthentication`/`OSStatus` in any signature: the `KeychainError` enum, the `KeychainNamespace` enum (`secret`/`oauth`), the `KeychainBackend`/`AuthSession` protocols (namespace-aware `add`/`read`/`delete`/`read(using:)` plus the **session-aware** `update(key:secret:using:namespace:)` in-place rotation write-back and the session-aware `list(using:namespace:)` name enumeration: a bare metadata listing (no values, no decryption) that never prompts on its own and rides the caller's single approval — there are **no** `exists` probes; auto-classify was removed, so namespace is explicit in the argument), the testable `SecretManager` orchestration (upsert ordering, read-before-delete; its `namespace` defaults to `.secret`; and `list(reason:)`, whose **authenticate-first** ordering **is** the gate — a bare metadata enumeration never decrypts so the item ACL is never evaluated, so `authenticate(reason:)` runs *before* any name is read and a cancelled/failed approval discloses nothing — then returns the names `.sorted()` for deterministic output), the `storeSecret(_:name:in:backend:)` write seam (a plain upsert: a write-time NUL guard — `guard !secret.contains(0)` — runs **first**, refusing to store a value with an embedded NUL that `get`/`run`'s `decodeEnvValue` would reject on read, so a bad value triggers no keychain I/O; then it delegates straight to the namespaced upsert — there is **no** cross-namespace probe, so the same name may be stored in both the `.secret` and `.oauth` stores), and the `decodeSecret`/`decodeEnvValue` UTF-8/NUL helpers (the read-time NUL check now defense-in-depth for legacy/out-of-band bytes the write-time guard never saw). Compiled into the app **and** the test bundle.
- `keymaster/keymaster/SystemKeychain.swift` — app-target only: the real `KeychainBackend` adapter wrapping the `SecItem*`/`LAContext` syscalls, the access-control/access-group constants (`makeAccessControl()` builds the sole `SecAccessControl` with `[.biometryAny, .or, .companion]` — Touch ID **or** a paired Apple Watch, still no `.userPresence`/`.devicePasscode` fallback and no reuse window), the `KeychainNamespace` → service-prefix mapping (`dev.mnck.` / `dev.mnck.oauth.`) and matching per-namespace account mapping (`keymaster` / `keymaster.oauth`, distinct so the prefix-overlapping namespaces can never alias), the `OSStatus` → `KeychainError` mapping, and the `list(using:namespace:)` name enumeration (a **fresh** query, **not** the single-key `baseQuery`: `kSecClassGenericPassword` + the per-namespace `kSecAttrAccount` + `kSecMatchLimitAll` + `kSecReturnAttributes:true`/`kSecReturnData:false` so nothing is decrypted, with `kSecUseDataProtectionKeychain:true` kept — without it the query hits the legacy file keychain and silently returns `[]` — and the session's `kSecUseAuthenticationContext`; `errSecItemNotFound` → `[]`, then it keeps only items whose `kSecAttrService` carries the namespace's own `servicePrefix` and strips that prefix to recover each name, so a plain key literally named `oauth.Foo` round-trips). Unit-untestable (needs signing + Touch ID hardware), so it is covered by the manual checklist below.
- `keymaster/keymaster/RunSupport.swift` — the pure, biometric-free helpers for the `run` subcommand **and** the namespaced-key parsing shared with top-level `get`: `parseNamespacedKey` (the single owner of the `secret.NAME`/`oauth.NAME` syntax — splits on the **first** `.`, prefix must be `secret` or `oauth`, key non-empty; throws `NamespacedKeyError` so a malformed key is rejected before any prompt, and is used by both `get` and `parseKeyMapping`), the namespace-carrying `KeyMapping` (`let namespace: KeychainNamespace` — the synthesized memberwise `init(env:key:namespace:)` with deliberately **no** default, so a missing namespace is a compile error rather than a silent `.secret`), `parseKeyMapping` (split on first `=`, parse the key-spec via `parseNamespacedKey`, derive env from the de-prefixed key or the explicit `ENV=` side), `mergedEnvironment`, plus the exec-input shapers `execArgv` (→ `["/usr/bin/env", "--"] + command`, the `--` keeping a dash-leading program name from being parsed as an `env` option) and `execEnvironmentBlock` (→ sorted `"KEY=VALUE"` strings; the `.sorted()` is **only** for deterministic test assertions — `execve`'s `envp` is unordered)), also Foundation-only and unit-tested headlessly (`KeychainNamespace` is in scope here with no `import` — same app target + same test bundle). The actual process replacement lives in `keymasterApp.swift`'s `execCommand` (an app-only `execve` syscall wrapper, **not** in the test bundle — like `SystemKeychain.swift`/`URLSessionTokenExchanger.swift` it is covered by the manual checklist, while its pure inputs `execArgv`/`execEnvironmentBlock` are unit-tested here).
- `keymaster/keymaster/TerminalInput.swift` — **Foundation-only** pure no-echo line-editing reducer extracted from the TTY reader: the `NoEchoLineEvent` enum (`continue`/`endLine`/`endInput`) and `noEchoLineEvent(after:into:)` (LF/CR → `endLine`; `0x04` Ctrl-D → `endInput` if the buffer is empty else `endLine`; `0x7F`/`0x08` → `eraseLastScalar` (the last UTF-8 scalar: pop the last byte, then keep popping continuation bytes `b & 0xC0 == 0x80` up to a 4-byte cap, so a backspace inside a multi-byte character never leaves a dangling partial sequence that would decode to U+FFFD); `0x15` Ctrl-U → kill the whole line (canonical VKILL); `0x17` Ctrl-W → `eraseLastWord` (canonical VWERASE: drop trailing blanks, then back to the previous SPACE/TAB or line start — scalar-safe because blanks are ASCII and never appear inside a multi-byte sequence); else append). Ctrl-U/Ctrl-W are re-implemented here because clearing `ICANON` (to drop the `MAX_CANON` cap) also disables the kernel's VKILL/VWERASE editing, so without these cases those bytes would be silently appended into the echo-off secret. The termios setup + per-byte `read()` loop in `readLineWithEchoOff` is the thin, manual syscall wrapper around this reducer; the reducer holds the unit-tested logic. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/OAuthRecord.swift` — **Foundation-only** OAuth data model: the tolerant `Codable` `OAuthRecord` (snake_case `token_endpoint`/`client_id`/`client_secret?`/`refresh_token`/`scopes?`) with `encoded()`/`validate()` (required fields non-empty, no field may contain an embedded NUL byte, and `token_endpoint` must be `https`; the NUL check closes the escaped-NUL bypass — `JSONEncoder` escapes U+0000 to a six-character text escape, so `storeSecret`'s byte-level `contains(0)` guard never sees it — that would otherwise let `oauth set` persist a silently-bricked credential), and the `TokenResponse` reply type (only `access_token` + optional `refresh_token`; the unused `token_type`/`expires_in` were dropped so a non-conformant value for either — e.g. a string `expires_in` — can no longer break the decode, and a custom `init(from:)` decodes `refresh_token` **tolerantly** — a present-but-non-string value, e.g. a JSON number, is treated as "no rotation" rather than failing the whole decode, so it can't mask a valid `access_token` either; `access_token` stays strictly required). Compiled into the app **and** the test bundle.
- `keymaster/keymaster/TokenExchanger.swift` — **Foundation-only** token-exchange seam: the `TokenExchanger` protocol plus the pure `buildTokenRequest` (form-encoded RFC 6749 §6 POST) and `parseTokenResponse` (2xx decode of `access_token` + optional `refresh_token` — unknown keys like `token_type`/`expires_in` are ignored — with a non-empty `access_token` check and a **NUL-byte rejection on both** `access_token` and the rotated `refresh_token` (the latter symmetric so a NUL credential can never be persisted by the rotation write-back, which bypasses `storeSecret`'s guard and would survive it anyway since `JSONSerialization` escapes U+0000); RFC `{error,error_description}` mapping incl. the `invalid_grant` re-auth hint) helpers. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/OAuthManager.swift` — **Foundation-only** OAuth orchestration shared by `get` and `run`: `OAuthManager` over the `KeychainBackend`+`TokenExchanger` seams. `get` and `run` funnel through one **authenticate-first** `resolve` core that calls `authenticate(reason:)` **once** (the single Touch ID prompt) and then does everything else through that session — there is **no** classify step: each `KeyMapping` carries its `secret.`/`oauth.` namespace (parsed by `parseNamespacedKey` before any prompt), so the loop **switches on `mapping.namespace`** — `.secret` keys are read+decoded (`read(using:)` → `decodeEnvValue`), `.oauth` keys are minted — and the rotation **write-back** (`update(using:)`) rides the same approval. The two wrappers are `resolveRunEnvironment(mappings:reason:)` (the `run` batch → `(env, staleKeys)`) and `resolveSecret(name:namespace:reason:)` (the `get` single name → `(value, refreshTokenStale)`, building one `KeyMapping(env: name, key: name, namespace: namespace)`); `mint(name:using:)` reads+exchanges an OAuth record under the supplied session, and `finish(name:recordData:session:)` decodes → exchanges → conditionally writes back. The write-back re-asserts the NUL guard at its own boundary — a rotated `refresh_token` containing a NUL is **not** persisted (skipped, flagged stale, the just-minted token still used), defense-in-depth in case a `TokenExchanger` conformer other than the shipping one doesn't reject it first. The rotation write-back is **lossless**: it edits `refresh_token` in place in the raw stored JSON object (`JSONSerialization`) rather than re-encoding the 5-field `OAuthRecord` model, so any extra keys an out-of-band write stored survive a rotation (the rotated bytes are decode-equivalent, not byte-canonical — `JSONSerialization` escapes the `token_endpoint` slashes). A name **absent in its declared namespace** surfaces a key-prefixed not-found from the authenticated read/mint (after the single prompt), so the missing-key behavior is unchanged — it just comes from the read rather than a removed classifier, and the name's absence is still never disclosed without a biometric approval first. The `MintResult` type carries the minted access token plus the stale-refresh-token flag. Compiled into the app **and** the test bundle.
- `keymaster/keymaster/URLSessionTokenExchanger.swift` — app-target only: the real `TokenExchanger` doing the synchronous `URLSession` POST (semaphore-gated, with a timeout) and casting to `HTTPURLResponse` for `parseTokenResponse`. It uses a **dedicated** `URLSession` (not `URLSession.shared`) with a no-redirect `URLSessionTaskDelegate` that **refuses every HTTP redirect** (returns `nil` from `willPerformHTTPRedirection`), so the credential-bearing POST body (the `refresh_token`, and any `client_secret`) is never resent to a redirect target; a redirecting endpoint surfaces as a request error. Unit-untestable (needs the network), so — like `SystemKeychain.swift` — it stays out of the test bundle and is covered by the live-provider smoke test below.

## Build

Open `keymaster/keymaster.xcodeproj` in Xcode, select the **keymaster** target ▸ **Signing & Capabilities**, set your **Team** (automatic signing; the Keychain Sharing capability + group `dev.mnck.keymaster` are already configured), and build (⌘B). The CLI binary is inside the bundle at `Keymaster.app/Contents/MacOS/keymaster`.

### Dependency

Argument parsing uses [swift-argument-parser](https://github.com/apple/swift-argument-parser) — each subcommand (the top-level `get`/`run` plus the `secret set`/`get`/`rm`/`ls` and `oauth set`/`get`/`rm`/`ls` groups) is a `ParsableCommand` in `keymasterApp.swift`; ArgumentParser is imported only there (the `SecretManager`/`SystemKeychain` layers are ArgumentParser-free). It is wired in as an SPM package and **resolved automatically**: Xcode fetches it on first build, and the release workflow's `xcodebuild archive` fetches it on a clean runner (no `-disableAutomaticPackageResolution`). The pinned version lives in `keymaster.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved`, which is **committed** so CI builds are reproducible rather than re-resolving to the latest `1.x`. The library links statically into the binary, so there is no extra framework to sign or notarize. If you ever re-add the package by hand, link **only** the `ArgumentParser` library product to the **keymaster** target — leave the `generate-manual` and `generate-docc-reference` executable products set to **None**.

A plain `swiftc keymaster*.swift` build will **not** work: the biometric `kSecAttrAccessControl` requires the `keychain-access-groups` entitlement, an unsigned binary that adds without it gets `errSecMissingEntitlement` (-34018), and a binary that carries it without a provisioning profile is SIGKILLed by AMFI. The `.app` target is what supplies the profile.

## Testing

The raw keychain/biometric/exec/network syscalls (`SystemKeychain.swift`, `URLSessionTokenExchanger.swift`) can't run headless or in CI — verify them by hand on a Touch ID Mac (below). The Foundation-only logic in `SecretManager.swift`, `RunSupport.swift`, `OAuthRecord.swift`, `TokenExchanger.swift`, `OAuthManager.swift`, and `TerminalInput.swift` **is** unit-tested via the `keymasterTests` target (Swift Testing — `import Testing`, `@Test`/`#expect`). `SecretManager`'s ordering and decode invariants (upsert `add`→read→delete→add, read-before-delete on `remove`, the authenticate-before-list `ls` flow — sorted/namespace-scoped output, empty store, single-session reuse, and the no-disclosure-on-auth-failure invariant — the `run` batch flow, UTF-8/NUL rejection) are exercised against a `FakeKeychainBackend` (a namespace-aware, dictionary-backed `KeychainBackend` that records call order — including `listUsing` — fakes `list(using:namespace:)` (returning the namespace's keys unsorted so the orchestration's sort is load-bearing), and can throw programmed `KeychainError`s); the OAuth mint/rotation/run-resolver flow (now switching on each mapping's explicit namespace, no classify) is exercised against the same fake plus a `FakeTokenExchanger` (a programmable `TokenResponse`-or-error double); and `RunSupport`'s pure helpers are tested directly — `parseNamespacedKey` (each row of the `secret.`/`oauth.` semantics table: success cases + missing/unknown-namespace + empty-key), `parseKeyMapping` (`secret.NAME`/`ENV=oauth.key`, the de-prefixed-env assertion, empty-env rejection), `mergedEnvironment` (override-wins merge), and the exec-input shapers `execArgv` (the `/usr/bin/env --` prefix + order preserved, incl. a dash-leading program name) and `execEnvironmentBlock` (sorted `KEY=VALUE`, a value containing `=`, empty dict → empty array). These cover only the *input shape*; the exec **behavior** (exit-code/signal propagation, env actually reaching the child, PATH resolution, the `127` path) lived in the old `runProcess` subprocess tests and, since `execve` cannot run headless, moved to the manual checklist when `run` switched from `Process` to exec. So the security-critical logic is covered without any real Keychain or network access:

```bash
xcodebuild test -project keymaster/keymaster.xcodeproj -scheme keymaster \
  -destination 'platform=macOS' -only-testing:keymasterTests
```

The `-only-testing:keymasterTests` scope is required: it excludes `keymasterUITests`, which launches the CLI app that `exit()`s immediately and would otherwise hang/fail the run. The `keymasterTests` bundle is **host-less** (`TEST_HOST`/`BUNDLE_LOADER` cleared) because the `@main` CLI exits before an app-hosted runner can attach; the Foundation-only files — `RunSupport.swift`, `SecretManager.swift`, `OAuthRecord.swift`, `TokenExchanger.swift`, `OAuthManager.swift`, and `TerminalInput.swift` — are compiled into the bundle via synchronized-group membership exceptions (there is no `@testable import`), which is why each is kept Foundation-only with no `@main`/ArgumentParser/keychain (`Security`/`LocalAuthentication`)/`URLSession`-syscall symbols. `SystemKeychain.swift` and `URLSessionTokenExchanger.swift` are deliberately **not** added to the membership exceptions — they hold the un-testable syscalls (keychain/biometric and network, respectively) and stay out of the test bundle.

Manual checks (need a physical Touch ID Mac and the signed `.app`):

- `secret set <newkey>`: stores the secret, **no** Touch ID prompt (the ACL is evaluated on access, not on creation).
- `secret get <key>`: a pure plain read — the Keychain challenges Touch ID; the prompt names the key. A name that is actually an OAuth record returns a plain-namespace "not found" (no mint, no OAuth lookup).
- `get secret.<key>` / `get oauth.<key>` (the **namespace-explicit** top-level resolver): **exactly one** prompt naming the de-prefixed bare key; `get secret.<key>` prints the plain value, `get oauth.<key>` prints a freshly-minted access token. `get <key>` with no prefix (and `get bogus.<key>` / `get secret.`) is rejected by `validate()` **before** any prompt with the new error text.
- `secret rm <key>` and overwrite (`secret set` on an existing key): both prompt. `SecItemDelete` does not decrypt the item, so the biometric ACL would not challenge it on its own — `SecretManager` forces an authenticated `backend.read(...)` first and proceeds to `delete` only if it succeeds (this read-before-delete ordering is the invariant covered by `SecretManagerTests`).
- `run --key secret.A --key TOKEN=oauth.B -- <command>`: **exactly one** Touch ID prompt, whose text names every key and the program, unlocks all the secrets. A `--key` with no/unknown namespace prefix is rejected by `validate()` **before** any prompt; a name absent in its declared namespace aborts **after** the single prompt (but before exec), naming the key and running nothing; cancelling the prompt also runs nothing. After the prompt keymaster **replaces itself** with the command via `execve` (through `/usr/bin/env`), so:
  - **interactive TUIs work** — `run --key <real-key> -- vim` (and `-- claude`, `-- pi`, `-- less`) run interactively and accept keyboard input under the single prompt, because the command inherits keymaster's PID, foreground process group, and the real controlling terminal on stdin/stdout/stderr (a `sh -c 'echo in=$([ -t 0 ]&&echo tty||echo no) out=$([ -t 1 ]&&echo tty||echo no); ps -o pgid,tpgid -p $$'` shows `in=tty out=tty` and `pgid == tpgid`). This is the bug this exec rewrite fixed; the old `Process`-based spawn put the child in a non-foreground group with a non-tty stdout, so it never got input.
  - **exit/signal status propagates for free** — the command's exit code is keymaster's exit code (`run --key <k> -- sh -c 'exit 7'` → shell `$?` is 7); a signal kills the (now command's) process directly, so the shell sees a **real signal death**, not a synthesized `128 + signo`.
  - **missing program** is reported by `/usr/bin/env` itself (its own not-found message, exit 127). `execCommand`'s own `Error: could not run command…` + exit 127 only fires if `execve("/usr/bin/env", …)` itself fails (env missing/unexecutable, near-impossible on macOS) or a `strdup` allocation fails.
  - **SIGPIPE** stays at its default disposition in the child (a `SIG_IGN` disposition would be inherited across exec, but `run` sets none — the `ls`-only `signal(SIGPIPE, SIG_IGN)` lives in `printNames`, which never runs in the `run` path), so a child writing to a closed pipe behaves as it would run directly.
  - (Exit-code forwarding / signal mapping / PATH resolution / the `127` path were **unit-tested** when `run` spawned via `Process`; with exec they are produced by the kernel and `/usr/bin/env`, not by keymaster code, so that behavioral coverage moved here to the manual checklist. The pure exec inputs `execArgv`/`execEnvironmentBlock` remain unit-tested.)
- `secret ls` / `oauth ls`: each prompts **exactly once** (Touch ID or Apple Watch) and prints its own namespace's names, one per line, sorted (`secret ls` shows plain secrets only, `oauth ls` OAuth records only — a plain key named `oauth.Foo` shows under `secret ls`, never `oauth ls`). An empty namespace still prompts and prints nothing (exit 0). **Cancelling the prompt lists nothing** — no name is disclosed. The gate is the explicit `authenticate(reason:)`: a `kSecReturnData:false` metadata enumeration never decrypts an item, so the ACL would never challenge the listing on its own.
- **Apple Watch approval (needs a paired, unlocked, on-wrist watch):** on any prompt (`secret get`/`secret rm`/overwrite `secret set`/`secret ls`/`oauth ls`/top-level `get`/`run`/`oauth get`/minting `get`) a nearby unlocked paired Apple Watch can approve via a **side-button double-click** in place of Touch ID — the ACL is `[.biometryAny, .or, .companion]`. With the watch locked / off-wrist / out of range the prompt silently falls back to Touch ID only (no spurious abort). **Items created before this ACL change keep their Touch-ID-only ACL** (the ACL is baked in at creation and never rewritten on read) until re-stored via `secret set`/`oauth set`; re-`secret set` any test keys to pick up the watch option.

OAuth manual checks (also need the network for the live exchange):

- for a minting `get`/`run`, the `SecItemUpdate` rotation write-back rides the single `authenticate` approval (it attaches `kSecUseAuthenticationContext`), so it adds no Touch ID prompt and the write-back stays atomic. (There is no namespace classify probe any more — the namespace is explicit in the `secret.`/`oauth.` prefix.)
- OAuth-namespace reads **do** prompt — **exactly once** each: `oauth get <name>` (a pure read), `keymaster get oauth.<name>` (minting: read + rotation write-back all under the one prompt), and `run --key …=oauth.<name>`.
- `oauth set <name>` interactive prompts: no-echo on `client_secret` + `refresh_token`, echo for the rest; the no-echo fields read in **non-canonical** mode, so a pasted credential **over 1024 bytes** is no longer beeped/truncated (the `MAX_CANON` line limit is gone), while backspace (DEL/Ctrl-H), **Ctrl-U (kill line)**, and **Ctrl-W (erase word)** all still edit and Ctrl-C still aborts (the reducer re-implements VKILL/VWERASE that clearing `ICANON` would otherwise have dropped). The non-TTY `OAuthRecord`-JSON-from-stdin path works.
- suspend/resume a no-echo prompt: at a `secret set` or `oauth set` no-echo prompt, press **Ctrl-Z** to suspend then `fg` to resume — the prompt **resumes** (the signal-interrupted `read()` retries on `EINTR`) rather than aborting with "could not read terminal input". (Only the EINTR retry is restored here; re-asserting raw/no-echo termios across the suspend is out of scope, matching `readLine()`'s behavior.)
- `oauth set` over an **existing OAuth record** prompts Touch ID (the upsert's read-before-overwrite).
- `keymaster get oauth.<name>` prints **only** the token on stdout (so `$(keymaster get oauth.X)` is clean); the stale-refresh-token warning goes to stderr.
- `keymaster run --key secret.A --key B=oauth.rec -- cmd` with a mixed plain+OAuth batch unlocks under **exactly one** Touch ID prompt naming all keys + the program; a name absent in its declared namespace is caught by the authenticated read/mint, so it deliberately aborts **after** that single prompt — naming the key and running nothing. Aborting *after* the prompt is intended: a name's existence is never disclosed without a biometric approval first. A `--key` with no/unknown namespace prefix is rejected by `validate()` before any prompt.
- duplicate names across namespaces **coexist** (the old cross-namespace refusal is gone): `printf %s "$SECRET" | keymaster secret set Dup` then `printf '%s' '{…}' | keymaster oauth set Dup` (and vice-versa) **both succeed** — no refusal, no `y/N`, no Touch ID on create — leaving `Dup` in both stores. `get secret.Dup` then reads the plain value and `get oauth.Dup` mints a token, each prompting exactly once. Behaves identically on a TTY and piped.
- `keymaster get <prefix>.<name>` error paths are **key-prefixed by the de-prefixed bare key** by design: `get oauth.<name>` over an OAuth record with a revoked refresh token (and `get secret.<name>` over a plain secret whose stored bytes fail to decode) prints an error beginning `"<name>: …"` to stderr (the bare key, not `oauth.<name>`), matching `run`'s resolver prefix. The deliberate, uniform `"<name>: …"` wording (incl. the post-prompt "not found" when a name is absent in its declared namespace) is intentional so a script can attribute the failure to the key it asked for.
- a rotating provider's new `refresh_token` is persisted (a second `run`/`get` still works); an expired/revoked refresh token surfaces the `invalid_grant` → "re-run oauth set" message.
- a token endpoint that responds with a 3xx redirect is **not** followed: the credential-bearing POST body (the `refresh_token`, and any `client_secret`) is **not** re-sent to the redirect target, and the request surfaces as an error (`get`/`run` fail) rather than minting a token.

Items live in the data-protection keychain (`kSecUseDataProtectionKeychain`), scoped to the access group from the entitlement. The legacy `security` CLI cannot see them, so a `security find-generic-password` "item not found" is expected and is **not** evidence of a bypass; the meaningful check is that `get` (and OAuth minting) requires Touch ID.

## Usage

The secret is read from **stdin**, never passed as an argument (an argument would leak via `ps` and shell history, CWE-214):

```
keymaster secret set <key>     # store a plain secret read from stdin; prompts on overwrite, not on first create
keymaster secret get <key>     # retrieve a plain secret only (no minting, no OAuth lookup), gated by Touch ID
keymaster secret rm <key>      # remove a plain secret, gated by Touch ID
keymaster secret ls            # list stored plain-secret names (one per line, sorted), gated by Touch ID
keymaster get secret.<key>     # retrieve a plain secret, gated by Touch ID (explicit namespace prefix)
keymaster get oauth.<name>     # mint an access token for an OAuth record, gated by Touch ID (explicit namespace prefix)
keymaster run --key <secret.NAME|oauth.NAME|ENV=secret.key|ENV=oauth.key> ... -- <command [args...]>  # inject secrets/tokens, one Touch ID prompt
keymaster oauth set <name>     # store an OAuth refresh-token record (prompts on a TTY, OAuthRecord JSON on stdin otherwise)
keymaster oauth get <name>     # print the stored OAuth record's JSON, gated by Touch ID
keymaster oauth rm <name>      # remove an OAuth record, gated by Touch ID
keymaster oauth ls             # list stored OAuth record names (one per line, sorted), gated by Touch ID
```

```bash
printf %s "$SECRET" | keymaster secret set MyKeyName   # piped
keymaster secret set MyKeyName                          # interactive no-echo prompt
```

The interactive no-echo prompt (used by `secret set` and by `oauth set`'s `client_secret`/`refresh_token` fields) reads in **non-canonical** mode (echo and `ICANON` both off), so there is no `MAX_CANON` 1024-byte line cap — a long pasted secret/credential is entered in full rather than beeped and silently truncated.

`secret ls`/`oauth ls` list the names stored in each namespace (one per line, `.sorted()`), gated by a **single** Touch ID / Apple Watch approval: `SecretManager.list(reason:)` authenticates *before* enumerating, because a `kSecReturnData:false` metadata query never decrypts and so never challenges the ACL on its own — cancelling the prompt discloses nothing, and an empty namespace prints nothing.

`run` injects one or more keychain secrets into a child process as environment variables, unlocking the whole batch with a single Touch ID prompt. Each `--key` carries an **explicit namespace prefix**: `--key secret.NAME` injects env var `NAME` from plain key `NAME`, `--key oauth.NAME` injects env var `NAME` from a freshly-minted token for OAuth record `NAME` (env is the **de-prefixed** key); `--key ENVNAME=secret.key`/`--key ENVNAME=oauth.key` injects env var `ENVNAME` from key `key` in that namespace (split on the first `=` only, then the right side is parsed for the `secret.`/`oauth.` prefix). The prefix is parsed and validated up front, so a missing/unknown-prefix `--key` is rejected **before** any prompt or exec. The resolver then authenticates once and, switching on each mapping's namespace, reads `.secret` keys and mints `.oauth` keys under that one approval — a name absent in its declared namespace aborts the batch naming the key (a deliberate post-prompt "not found", so the name's existence is never disclosed without a biometric approval first). Any missing/unreadable key aborts before the command runs. keymaster then **becomes** the command via `execve` (through `/usr/bin/env`), so the command inherits keymaster's PID/foreground process group/controlling terminal — interactive TUIs work — and exit/signal status propagates directly: the command's exit code is keymaster's, and a signal terminates the process for real (a true signal death, not a synthesized `128 + signo`).

### OAuth refresh-token records

An OAuth record stores a long-lived refresh-token credential (behind Touch ID) and `get`/`run` exchange it for a short-lived **access token** on demand via the RFC 6749 §6 refresh-token grant — no manual copying of expiring tokens.

```bash
# store a record: on a TTY each field is prompted (no-echo for client_secret + refresh_token); otherwise read OAuthRecord JSON from stdin
keymaster oauth set GitHub
printf '%s' '{"token_endpoint":"https://example.com/token","client_id":"abc","refresh_token":"r0","scopes":"repo"}' | keymaster oauth set GitHub

keymaster get oauth.GitHub                 # Touch ID → mints + prints ONLY the access token on stdout (warnings go to stderr)
keymaster run --key TOKEN=oauth.GitHub -- ./deploy  # injects a freshly-minted TOKEN, one prompt
```

The `token_endpoint`, `client_id`, and `refresh_token` fields are required (validated on `oauth set`: non-empty, and `token_endpoint` must be `https`); no field may contain an embedded NUL byte (a JSON-escaped NUL slips past the raw-byte store guard, so `validate()` rejects it explicitly); `client_secret` (omit/empty = public client) and `scopes` are optional. keymaster never caches an access token — it mints fresh each time. If the provider rotates the refresh token (returns a new `refresh_token`), the stored record is updated in place atomically; the write-back is **lossless** — it edits `refresh_token` in the raw stored JSON rather than re-encoding the modelled fields, so any extra keys survive (the rotated bytes are decode-equivalent, not byte-canonical). If that write-back fails, the just-minted token is still used and a warning naming the key (re-run `oauth set`) is printed to **stderr** (non-fatal). An expired/revoked refresh token surfaces an `invalid_grant` → "re-run oauth set" message. The token exchange uses a dedicated session that **refuses HTTP redirects**, so the credential-bearing POST is never resent to a redirect target (a redirecting endpoint fails the request instead).

## Storage

Plain secrets are stored in the data-protection keychain under service `dev.mnck.<key>`, account `keymaster`, access group `<TeamID>.dev.mnck.keymaster`, accessibility `kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly`, and a `[.biometryAny, .or, .companion]` access control (Touch ID **or** a paired Apple Watch; deliberately **no** `.userPresence`/`.devicePasscode` fallback). The ACL is baked into each item at creation and is never rewritten on read, so items stored before this watch-capable ACL keep their original Touch-ID-only ACL until re-stored via `secret set`/`oauth set` (informational; decode-irrelevant).

OAuth records live in a **separate namespace** under service `dev.mnck.oauth.<name>` and a **distinct account** `keymaster.oauth` (same access group / accessibility / access control), holding the whole credential as canonical JSON. The distinct account matters: the plain service prefix `dev.mnck.` is a textual prefix of the OAuth one `dev.mnck.oauth.`, so a plain key like `oauth.<name>` would otherwise compute the same `kSecAttrService` as OAuth record `<name>`; since the keychain primary key is the (service, account) pair, the per-namespace account keeps the two stores **physically disjoint** for **all** key text. The two stores are **independent**: the same name may live in **both** of them at once (a plain secret `Foo` and an OAuth record `Foo` are two separate keychain items). `secret set`/`oauth set` are plain upserts — neither consults the other namespace, so creating a duplicate across namespaces is allowed and silent (no refusal, no `y/N`, no Touch ID on create). Retrieval is **unambiguous because it is explicit**: `get`/`run` take a `secret.`/`oauth.` prefix (parsed by `parseNamespacedKey` before any prompt) and read or mint strictly in that named namespace — there is no auto-classify and no `.oauth`-wins tie-break, so `get secret.Foo` always reads the plain secret and `get oauth.Foo` always mints from the OAuth record, even when both exist.
